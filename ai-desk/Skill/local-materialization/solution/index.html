Production Solution

AI Desk Skills 本地物化与 Runtime 原生发现上线方案

方案目标是让 AI Desk 已安装 skills 以标准 project skills 的形式被 Codex、Claude Code、Cursor、GitHub Copilot 等 runtime 原生发现和使用。用户不需要通过 slash 显式拼接平台 MCP 指令，runtime 可以在会话初始化后基于 skill name、description 和 skill 内部引用自然触发。

上线方案 更新时间：2026-07-08 Scope：Local runtime task workspace Core：~/.aidesktop/skills + provider symlinks

目标

1

显式自然语言使用

用户输入“请使用 bug-analysis 这个 skill 帮我分析下...”时，runtime 能识别到本地已安装的 bug-analysis skill，并按该 skill 的说明执行。

2

隐式意图触发

用户输入“帮我分析一下这个 bug”时，runtime 可以基于可见 skill 的 name 和 description，自动选择并使用 bug-analysis。

3

Skill 间自然引用

bug-analysis 的说明中提到需要使用 code-review 时，runtime 可以在同一个 project skills inventory 中自然定位并继续使用 code-review。

目标状态 AI Desk skills 对 runtime 的呈现方式与项目自带标准 skills 一致：启动时可见、自然语言可引用、skill 内部可链式引用、slash 只作为快捷选择入口而不是唯一触发机制。

整体方案

核心设计 用户已安装 skills 统一物化到 ~/.aidesktop/skills；每个 task workspace 只创建 runtime 专属入口目录，并通过 symlink 指向该全局目录。runtime 从自己的原生 project skills 入口加载 AI Desk skills。

Task workspace 目录

~/.aidesktop/workspace-7f3a9/

  .claude/skills -> ~/.aidesktop/skills
  .cursor/skills -> ~/.aidesktop/skills
  .github/skills -> ~/.aidesktop/skills
  Fiji/
    ...worktree files or project symlink...
路径原则 ~/.aidesktop/skills/{skillName}/SKILL.md 是 runtime 可见的稳定路径。版本更新时替换该稳定路径下的文件内容，不通过切换 symlink target 暴露版本目录。同名 skill 不建议通过路径或 name 后缀规避，应在安装或启用策略中避免同时暴露给 runtime。

本地目录规则

规则 说明
稳定目录 每个 installed skill 在 ~/.aidesktop/skills 下对应一个稳定目录。runtime 读取的是该稳定目录，不读取版本目录。
不直接写入业务 repo AI Desk task 会创建外层 ~/.aidesktop/workspace-xxxx，provider skills 入口放在该外层目录，不写入用户选择的 Fiji/ 等业务项目目录。

Task 创建时的 workspace 准备

  1. AI Desk 创建 task workspace，例如 ~/.aidesktop/workspace-7f3a9/。
  2. 将用户选择的项目放入或链接到 workspace，例如 ~/.aidesktop/workspace-7f3a9/Fiji/。
  3. 在 workspace 外层创建四个 provider skills 入口：.codex/skills、.claude/skills、.cursor/skills、.github/skills。
  4. 四个入口均 symlink 到 ~/.aidesktop/skills。
  5. runtime 以 ~/.aidesktop/workspace-7f3a9 作为可发现 skills 的工作目录启动。

模块一 Q&A

Q1. 为什么 symlink 方案可行？

Codex、Claude Code、Cursor、GitHub Copilot 均可以从各自 project skills 目录加载 symlink 指向的外部 skills。也就是说，AI Desk 不需要把所有 skill 文件复制到每个 task workspace，只要在 workspace 下提供 runtime 原生入口即可。

Q2. 为什么不只创建 .agents/skills？

各 runtime 对 project skills 入口目录的支持不完全一致，其中 Claude Code 不支持从 .agents/skills 加载 project skills。为了覆盖 Codex、Claude Code、Cursor、GitHub Copilot，需要分别创建 .codex/skills、.claude/skills、.cursor/skills、.github/skills。

Q3. 为什么不使用 ~/.aidesktop/skills/bug-analysis/1.0.0 这种版本目录作为 runtime 可见路径？

runtime 在同一 session 中可能缓存已使用 skill 的 realpath。若 bug-analysis 从 1.0.0 symlink 切到 1.1.0，Codex 仍可能继续读取旧版本目录。稳定路径原地替换可以避免该问题，因此版本号应记录在 manifest 中，而不是出现在 runtime 可见路径中。

Q4. 为什么优先保持目录名和 SKILL.md name 一致？

不同 runtime 对 skill 可调用 ID 的理解不同：有的更看重目录名，有的更看重 frontmatter name。保持二者一致，可以让用户自然语言提到 skill name、skill 内部引用其他 skill、runtime 展示 skill 列表时都使用同一个名称。

Q5. 为什么不建议在文件夹名或 skill name 后拼 id？

拼 id 后缀会让 runtime 可见名称偏离用户和 skill 内容里使用的真实名称。例如用户说 bug-analysis，但本地实际暴露的是 bug-analysis-6f2a9c1e，runtime 可能无法稳定对应。更关键的是，实验 4 中 Cursor 在“一个 skill 间接引用另一个 skill”的场景下，目录名与 SKILL.md name 不一致时没有成功触发链式 skill；这说明加后缀会破坏 skill 间按 name 自然引用的可靠性。上线方案因此不建议通过拼 id 解决同名问题，而应优先从 AI Desk 的安装或启用策略上避免同名 skill 同时暴露给 runtime。

模块二：Sync 方案

同步触发点

触发点 动作 结果
FE 启动并连接 daemon 进入统一 Skill Sync Pipeline：拉取服务端已安装清单，读取本机已物化清单，并对比出新增、升级、删除。 本地 ~/.aidesktop/skills 在应用启动后快速收敛到服务端状态。
BE socket 推送 收到 skill install、uninstall、version changed 事件后，进入同一条统一 Skill Sync Pipeline。 用户在 AI Desk 更新 skill 后，本地目录自动更新。
用户点击 install / uninstall 服务端安装状态变更成功后，进入同一条统一 Skill Sync Pipeline。 用户主动安装或卸载 skill 后，本地物化目录与最新安装状态一致。

BE 接口

接口 返回或行为 说明
A. 获取用户已安装 Skill 清单 Array<{ id, name, version }>，可扩展 updatedAt、sha256。 id + version 是 diff 的最小依据；前提是同一个 id + version 内容不可变。
B. 下载指定 ids 的 skill bundles 返回完整文件树：SKILL.md、references、scripts、assets。 FE 只请求 diff 后需要新增或更新的 ids，避免全量下载。
C. Skill version socket 推送 skill install、uninstall、version changed。 socket 只负责通知；最终状态仍通过 API A + local manifest diff 收敛。

模块二 Q&A

Q1. 为什么用 id + version 做 diff？

Skill 有服务端版本语义，且更新会自动变更 version。在保证同一个 id + version 内容不可变的前提下，id + version 足以判断本地是否需要下载更新。sha256 可作为后续完整性校验字段追加。

Q2. 为什么 socket 后仍要走 diff？

socket 是通知通道，不应作为最终状态。FE 收到推送后重新拉取 manifest list，与本地 manifest diff，保证漏消息、重复消息、乱序消息都能通过最终状态收敛。

Q3. 为什么不在每次发送消息前做 blocking ensure？

Skill 文件同步是用户级状态，不是每轮消息的必需重操作。FE 启动同步和 socket 增量同步可以覆盖常规路径；task 创建时只需要建立 workspace symlink。后续如需要更强一致性，可在本地 manifest 缺失或过期时增加轻量 startup gate。

模块三：无 Worktree Task 的 Workspace 包装目录

当 task 创建时不使用 worktree，且用户选择的是已有本地项目目录，例如 FIJI，AI Desk 仍然需要创建一个自己管理的 workspace 外壳目录。真实项目通过子目录 symlink 暴露给 runtime，四个 provider skills 入口只创建在 workspace 外壳层，避免污染用户真实项目。

真实项目目录

/Users/karmiy.hong/Documents/Projects-stage-2/FIJI/
  package.json
  src/
  ...real project files...

AI Desk workspace 外壳

~/.aidesktop/workspace-7f3a9/
  .codex/skills  -> ~/.aidesktop/skills
  .claude/skills -> ~/.aidesktop/skills
  .cursor/skills -> ~/.aidesktop/skills
  .github/skills -> ~/.aidesktop/skills
  FIJI -> /Users/karmiy.hong/Documents/Projects-stage-2/FIJI
关键约束 runtime cwd 使用 ~/.aidesktop/workspace-7f3a9，而不是用户真实项目目录。这样 runtime 能在 workspace 外层发现 .codex/skills、.claude/skills、.cursor/skills、.github/skills，同时所有新增 runtime 入口都不会写进真实 FIJI 项目。

创建流程

  1. Task 创建时分配 workspace id，例如 workspace-7f3a9。
  2. 在 ~/.aidesktop/ 下创建 workspace-7f3a9/。
  3. 在 workspace 外层创建四个 provider skills symlink，统一指向 ~/.aidesktop/skills。
  4. 为用户选择的真实项目创建子目录 symlink，例如 workspace-7f3a9/FIJI -> /真实路径/FIJI。
  5. runtime 从 workspace 外层启动；需要访问代码时进入 FIJI/ 子目录。

为什么需要该包装目录

问题 处理方式
不使用 worktree 时，真实项目目录本身没有 AI Desk 管理的外层。 创建 ~/.aidesktop/workspace-xxxx 作为 AI Desk-owned runtime cwd。
如果直接在真实项目下创建 .codex/skills 等目录，会污染用户项目。 四个 provider skills 入口只创建在 workspace 外层；真实项目只作为子目录 symlink 出现。
runtime 需要同时看到 skills 和代码。 workspace 外层提供 skills 入口；项目子目录 symlink 提供代码访问。

职责边界

BE

  • 维护 skill install、权限、版本、文件内容。
  • 提供 manifest list、bundle download、socket notification。
  • 保证 id + version 内容不可变，提供本地同步所需的最终状态。

FE

  • 作为 sync coordinator，执行 startup diff 与 socket-triggered diff。
  • 调用 daemon 文件能力写入 ~/.aidesktop/skills。
  • task workspace 创建后建立 provider skills symlinks。

Daemon

  • 只提供通用文件系统能力：stat、read、write、rename、symlink。
  • 运行 runtime CLI 并保持统一 stream 事件输出。
  • 不访问 BE，不读取 DB，不理解 AI Desk skill 业务模型。

上线实现拆解

模块 工作项 验收点
Skill Manifest BE 提供用户 installed skills manifest list，支持 id、name、version。 FE 可以通过 id + version 和本地 manifest 判断需要下载、更新、删除的 skills。
Bundle Download BE 支持按 ids 下载完整 skill 文件树。 FE 可以只下载 diff 后变化的 skills。
Local Materialization FE 调 daemon 将 skill 文件写入 ~/.aidesktop/skills/{skillName}，并维护本地 manifest。 本地目录结构符合标准 skill 目录，目录名与 SKILL.md name 保持一致。
Workspace Symlinks task 创建 workspace 后建立四个 provider skills symlinks。 四个 runtime 新 session 均可发现 AI Desk installed skills。
No-worktree Workspace Wrapper 不使用 worktree 的 task 也创建 ~/.aidesktop/workspace-xxxx，并将真实项目作为子目录 symlink 放入。 provider skills 入口只存在于 AI Desk workspace 外层，不污染用户真实项目目录。
SlashPrompt Update slash 选择 skill 后不再拼 MCP 获取指令，而是以 runtime 原生 skill name 作为快捷引用。 slash 与自然语言路径共享同一套 runtime-visible skills。
Observability 记录 manifest diff、download count、write result、symlink result、skill name conflict。 上线后可以定位本地同步失败、workspace symlink 缺失和 runtime 未发现 skill 的问题。

上线验收标准

用户行为验收

  • 用户输入“请使用 bug-analysis skill...”时，runtime 使用 bug-analysis。
  • 用户输入“帮我分析一下这个 bug”时，runtime 能基于可见 skill 自主选择 bug-analysis。
  • bug-analysis 内部引用 code-review 时，runtime 能继续定位并使用 code-review。
  • 用户通过 slash 选择 skill 时，行为与自然语言使用同一套本地物化 skill。

工程行为验收

  • ~/.aidesktop/skills 中存在稳定 skill 目录和 manifest。
  • task workspace 外层存在四个 provider skills symlinks。
  • 不使用 worktree 的 task 仍创建 ~/.aidesktop/workspace-xxxx 外壳，真实项目以子目录 symlink 接入。
  • 真实项目目录下不会新增 .codex/skills、.claude/skills、.cursor/skills、.github/skills。
  • 版本更新后稳定路径文件内容更新，不出现 runtime-visible version directory symlink 切换。
  • daemon 不包含 AI Desk skill 业务逻辑，只执行通用文件操作。
